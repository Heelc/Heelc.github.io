Posted on 2018-10-15 | In Git , 笔记

Git？

Git是一个分布式版本管理系统，Git可以在任何时间点，把文档的状态作为更新记录保存起来。因此可以把编辑过的文档复原到以前的状态，也可以显示编辑前后的内容差异。
而且，编辑旧文件后，试图覆盖较新的文件的时候（即上传文件到服务器时），系统会发出警告，因此可以避免在无意中覆盖了他人的编辑内容。

回退版本

在Git中，当前版本为HEAD，上个版本为HEAD^,上上个版本为HEAD^^，过于久远的版本，比如往上的第10个版本，可以写为HEAD~10。
回退命令为git reset --hard <version>，比如回退到上个版本

回退到上个版本后再使用git log最新版本的日志已经消失不见了，那么怎么才能再回到最新版本呢？只要你的git bash还保留着最新版本的版本号，就能直接通过git reset --hard <version>回到该版本。
如果重启了git bash 怎么办呢？在git中你总是有后悔的办法，git reflog会记录你的每一次命令，在过往命令中找到你想回到的版本号吧，然后，时光倒流。

暂存区

通过git add <file>将你想进行版本管理的文件提交到暂存区，之后你对该文件的内容修改都会在暂存区中记录下来。当你觉得暂存区里的文件版本达到了预期的阶段，就可以通过git commit -m "msg"将暂存区里所有文件提交到版本库里的分支中，形成一个版本记录。当需要回滚版本的时候，就可以将分支中所保存的版本通过git reset --hard <version>回滚出来。

Git管理跟踪的是<修改>

Git的特别之处是它管理和跟踪的是修改，每当你对文件进行修改后，都要将该次修改的文件add至暂存区中，否则在commit到分支的时候不会提交这次的最新修改版本。
撤销修改$ git checkout -- readme.txt,删除文件git rm <file>

分支管理

并行开发，在不影响主分支的情况下有独属于你的分支。
Git的分支创建其实是新建了一个分支的指针例如b来指向当前master分支，切换分支的时候就将HEAD指向b。当修改完文件提交给版本库的时候，会将b分支的指针指向该新版本，指向当前分支的HEAD指针也会随之指向b分支，从而形成一条分支版本线。

分支合并和删除分支

合并分支，解决冲突

当master分支和副分支都有了新的提交，在合并的时候可能会出现冲突。
这时需要查看master分支内容，并手动修改冲突部分，然后再合并，最后删除副分支。

分支命令小结一下：

  • 查看分支：git branch
  • 创建分支：git branch <name>
  • 切换分支：git checkout <name>
  • 创建+切换分支：git checkout -b <name>
  • 合并某分支到当前分支：git merge <name>
  • 删除分支：git branch -d <name>

分支管理策略

在通常合并分支的时候，如果可能Git会优先使用Fast foward模式来合并分支，在这种模式下合并后，删除了分支，便会丢失以前分支的信息。通过强制禁用Fast foward模式，Git在git meger合并的时候就会生产一个新的commit，这样，从分支日志git log上就可以看到之前分支的信息。
例子：假设新创建了一个dev分支并已经修改文件。回到master分支准备进行合并.

-no-ff参数表示禁用Fast foward模式，由于本次合并要提交一个新的commit所以加上-m参数附上提交描述。

合并和可以用git log查看分支日志，可见分支都合并到了新增的commit上。

分支策略：实际开发中，master分支只用于发布版本，必须非常稳定，所以开发过程应该在附属分支上进行，当新功能开发完成后再合并到master分支上。所以在团队合作时，每个人都应该有独属于自己的分支，并行开发功能模块，互不影响。

分支场景

BUG 分支:
修复bug的任务在创建的临时分支BUG中进行，完成后合并到需要修复bug的分支上。
但是这时你如果正在开发一个新功能，进度还没完成，没法提交的时候怎么办？
使用Git提供的stash命令，将当前进度储藏起来，等以后恢复现场后继续工作。

现在用git status查看工作区，就是干净的。可以开始创建临时的BUG分支来修复bug了。
修复完bug，提交commit后，切换到需要修复bug的分支上合并临时BUG分支，然后删除BUG分支。
然后我们要回到之前的工作分支上了，刚才的工作现场存到stash里面去了，用git stash list查看

恢复工作现场的方法有两种：

  1. git stash apply <stash@{?}>应用储藏，应用后，stash内容不会删除，如果需要删除，你需要使用命令git stash drop <stash@{?}>
  2. git stash pop，把stash区看做栈，pop出最新的stash，出栈后自动删除了最新的stash。

Feature 分支:
开发新功能时最好都创建一个临时的分支来进行开发，但是如果想要丢弃这个还没有被合并的分支，可以通过
git branch -D <name>强行删除该分支。

多人协作

当你从远程仓库克隆到本地的时候，Git已经将本地的master分支和远程的master分支关联起来了，同时远程的仓库默认名是origin，要查看远程仓库的详细信息，用git remote -v

之后就能git pull成功了。

因此，多人协作的工作模式通常是这样：

  1. 首先，可以试图用git push origin <branch-name>推送自己的修改；
  2. 如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；
  3. 如果合并有冲突，则解决冲突，并在本地提交；
  4. 没有冲突或者解决掉冲突后，再用git push origin <branch-name>推送就能成功！

如果git pull提示no tracking information，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream-to <branch-name> origin/<branch-name>。
这就是多人协作的工作模式，一旦熟悉了，就非常简单。

小结

  • 查看远程库信息，使用git remote -v；
  • 本地新建的分支如果不推送到远程，对其他人就是不可见的；
  • 从本地推送分支，使用git push origin <branch-name>，如果推送失败，先用git pull抓取远程的新提交；
  • 在本地创建和远程分支对应的分支，使用git checkout -b <branch-name> origin/<branch-name>，本地和远程分支的名称最好一致；
  • 建立本地分支和远程分支的关联，使用git branch --set-upstream <branch-name> origin/<branch-name>；
  • 从远程抓取分支，使用git pull，如果有冲突，要先处理冲突。
